# search-flow.md

> **먼저 확인할 것**: `index.html` 전체를 검색해도 텍스트 검색 입력창이나 필터링 함수가 코드에 존재하지 않는다 (`검색`, `search`, `filter` 관련 코드 없음). 그런데 `docs/AI_CONTEXT.md`의 User 사이트 "주요 기능" 목록에는 **"검색"이 포함되어 있다.** 이는 문서와 실제 구현 사이의 괴리이며, 이 문서는 "프로젝트에 없는 기능은 작성하지 않는다"는 원칙에 따라 존재하지 않는 검색 기능의 내부 동작을 지어내지 않는다. 대신 실제로 존재하는, 검색과 가장 가까운 기능인 **탐색(browse) 흐름**을 정리했다.

---

# 실제로 존재하는 것: 탐색(browse) 흐름

검색창에 키워드를 입력하는 방식이 아니라, 아래 3단계 클릭으로 원하는 항목을 찾아가는 구조다.

```
1. 맵 선택 (#viewGrid, mapGrid)
   ↓ 맵 타일 클릭 → openMap(id, name)
2. 팀 선택 (RED / BLUE 토글, #viewDetail)
   ↓ setTeam('red' | 'blue') → renderCards()
3. 태그별 카드 목록 (맵 지명 / 위폭 / 팁, #cardGrid)
   ↓ 카드 클릭 → openOverlay(id) 로 영상/이미지 재생
```

- 텍스트 검색, 자동완성, 키워드 필터는 없다.
- 유일한 필터링은 `renderCards()`의 `items.filter(i => i.map_id === currentMap && i.team === currentTeam)` — 즉 "현재 선택한 맵 + 팀"으로만 좁혀진다.
- 태그 그룹핑(`groups`, `tagOrder`)은 검색이 아니라 같은 화면 안에서 항목을 분류해 보여주는 것에 가깝다.

---

# 문서와 구현이 다른 이유(확인된 사실만)

- `AI_CONTEXT.md`는 "검색"을 User 사이트의 주요 기능으로 나열하고 있다.
- 실제 `index.html`에는 검색 관련 UI/로직이 없다.
- 이 괴리가 (a) 아직 구현하지 않은 예정 기능을 문서에 먼저 적어둔 것인지, (b) 과거 어떤 버전에는 있었는지, (c) 단순히 "탐색"과 "검색"을 같은 의미로 서술한 것인지는 코드나 git history만으로는 판단할 수 없다. **추측하지 않고 사실만 기록한다.**
- `AI_CONTEXT.md`는 이 작업에서 수정하지 않았으므로, 실제 검색 기능을 붙일지 / 문서 표현을 정리할지는 사용자가 판단해야 한다.

# 만약 실제 텍스트 검색을 붙이게 된다면 (참고용 메모, 미구현)

현재 구조에서 검색을 추가한다면 손댈 지점은 다음과 같다 — **아직 구현되지 않았고, 아래는 코드 위치 안내일 뿐 설계 확정 사항이 아니다.**

- `renderMapGrid()` — 맵 이름 기준 필터
- `renderCards()`의 `filtered` 계산 — `it.title`/`it.note` 기준 필터 추가 지점
- 새 검색 기능을 추가하려면 CODING_RULES.md의 "새 라이브러리 추가 금지(사용자 승인 필요)" 원칙에 따라, 별도 검색 라이브러리 도입 여부를 먼저 사용자와 협의해야 한다.
