# SQL 실행 규칙 (2026-07-20 변경)

기존 "SQL은 작성만 하고 사용자가 직접 실행한다"는 규칙을 위험도 기준으로 세분화한다.

## 자동 실행 가능 (Supabase MCP 플러그인 사용)
- SELECT (조회)
- INSERT (데이터 추가)
- UPDATE (데이터 수정)

## 반드시 사용자 확인 후 실행
- DELETE (데이터 삭제)
- DROP / TRUNCATE (테이블·데이터 전체 삭제)
- CREATE TABLE (테이블 생성)
- ALTER TABLE (구조 변경, 컬럼 삭제 포함)
- RLS 정책 생성/변경/삭제

이 경우 실행 전 반드시 다음을 사용자에게 명시한다:
- 무엇을 실행하려는지
- 되돌릴 수 없는 작업임을 명확히 고지
- "진행할까요?" 확인을 받은 후에만 실행

## 이유
사용자가 비전공자로 SQL 자체의 위험도를 스스로 판단하기 어렵기 때문에,
이 구분과 판단은 항상 Claude Code/Claude Chat이 대신 수행한다.

## 말투 규칙

- 코드 주석, 커밋 메시지, 작업 중 스스로 판단을 남기는 혼잣말(예: "일단 이 함수부터 확인해보자" 같은 사고 과정)은 편한 톤으로 써도 된다.
- 하지만 사용자에게 직접 말을 거는 모든 응답 — 작업 결과 보고, 질문, 확인 요청, 에러 설명 등 — 은 항상 존댓말(정중한 톤)로 작성한다.
- 판단 기준: "이건 사용자가 읽으라고 쓰는 문장인가?"라고 자문해서, 그렇다면 존댓말을 쓴다.

## 커밋 전 필수 리뷰 규칙 (Mode A 기본)

이 프로젝트의 기본 협업 흐름은 Mode A(Claude Chat 설계 → Claude Code 구현 → Codex 리뷰(자동 발동 시 자동, 미발동 시 codex:rescue로 수동) → 커밋)이다.
지시서(프롬프트)에 "Codex 리뷰"에 대한 언급이 없더라도, 아래 절차를 항상 스스로 따른다 — 지시서에 없다는 이유로 리뷰를 건너뛰지 않는다.

1. 코드 변경을 완료한다 (아직 커밋하지 않는다).
2. 변경된 diff를 Codex에게 리뷰 요청한다.
3. Codex의 리뷰 결과(승인/지적 사항)를 받는다.
   - 지적 사항이 있으면 반영 후 다시 diff를 정리한다.
   - 승인되면 다음 단계로 진행한다.
4. 그 다음에만 git add / commit / push를 진행한다.

예외: 사용자가 명시적으로 "이번엔 리뷰 없이 바로 커밋해줘"라고 말한 경우에만 이 절차를 건너뛴다.
Claude Code 토큰이 부족해 Mode B(GPT 설계 → Codex 리뷰+구현+커밋)로 전환된 경우, 이 규칙은 GPT/Codex 쪽 워크플로우에 준하는 절차로 대체된다 (Claude Code가 직접 수행하는 경우가 아니므로).

이 예외는 사용자의 명시적 요청이 있을 때만 적용되며, 다음과 같은 경우에도 예외로 인정되지 않고 동일하게 리뷰를 거쳐야 한다:
- Mode A/B 전환이 진행 중인 경우
- 변경이 사소해 보이는 경우
- 이미 이전에 리뷰받은 코드를 그대로 재사용하는 경우

만약 실수로 리뷰 전에 커밋·push했다면, 그 시점에 바로 사후 리뷰를 받고 그 사실과 원인을 작업 보고서에 명시한다.

작업 보고 마지막에 항상 아래 항목을 포함한다:
"이번 작업에서 Codex 리뷰를 거쳤는지 여부, 거쳤다면 리뷰에서 나온 지적 사항이 있었는지" — 사전 리뷰가 아니라 사후 리뷰였다면 그 사실과 원인도 함께 적는다.

## 지시서 범위 밖 변경을 발견했을 때의 되돌림 기준

작업 중 지시서 범위 밖의 변경(코드/문서 무관)을 발견하면 다음 기준으로 판단한다.

- 판단 기준은 **"커밋됐는가"**이지 "푸시됐는가"가 아니다. 커밋은 이미 누군가 의도적으로 확정한 작업이라는 뜻이므로, 아직 푸시 전이라도 함부로 되돌리지 않는다.
- 그 변경이 **아직 커밋되지 않은 상태(uncommitted)**라면, 이번 작업과 무관하다고 판단 시 되돌려도 된다. 단, 되돌리기 전 발견 사실과 이유를 작업 보고서에 반드시 남긴다.
- 그 변경이 **이미 커밋된 상태**라면(푸시 여부 무관), 이번 작업과 무관해 보여도 **절대 되돌리거나 수정하지 않는다.** 다른 세션에서 사용자가 승인한 정당한 작업일 가능성이 높다.
- 커밋 여부가 애매하면 `git log`로 먼저 확인하고, 확인 없이 임의로 되돌리지 않는다.
- 이미 커밋된 범위 밖 변경이 정말 문제가 있어 보인다면(예: 명백한 실수), 되돌리지 말고 **사용자에게 알리고 판단을 구한다.**
- **커밋됐지만 푸시가 안 된 상태를 발견했다면**, 그 자체를 "이상한 상태"로 보지 말고, 단순히 아직 푸시 단계까지 못 간 정상적인 작업 흐름일 수 있음을 감안한다. 필요하다면 되돌리는 대신 **그대로 푸시를 마저 진행하거나, 사용자에게 푸시해도 되는지 확인**하는 쪽을 우선 고려한다.

## 작업 완료 보고 체크리스트 규칙

지시서에 "테스트 확인 사항" 목록이 있으면, 완료 보고 시 이 목록을 서술형으로 요약하지 않고 항목을 그대로 가져와 하나씩 표기한다.

- ✅ 확인함
- ❌ 누락되었거나 확인하지 못함
- ➖ 이번 작업과 무관해 해당 없음

❌로 표기한 항목이 있으면 왜 확인하지 못했는지 이유를 함께 적는다. 이유 없이 빠뜨리지 않는다.

예시(진영 선택 로직 지시서 기준):
- ✅ mTeam select 옵션이 RED/BLUE + "선택 안 함(공통)"만 있는지
- ✅ 추가 모달이 항상 미선택으로 시작하는지
- ❌ 미선택 저장 항목을 즐겨찾기 추가 시 FAVORITE에도 표시되는지 — 최초 보고에서 누락, 사용자 지적 후 추가로 확인함
