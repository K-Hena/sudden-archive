# 공통 작업 원칙

모든 작업에서 반드시 지켜야 할 사항

- AI_CONTEXT.md를 기준으로 작업한다.
- 기존 구조를 최대한 유지한다.
- 추측하지 않는다.
- 존재하지 않는 파일을 가정하지 않는다.
- 필요한 파일만 수정한다.
- 완료 후 영향 범위를 확인한다.
- 사용자 승인 없이 구조를 변경하지 않는다.

# 9. Codex 작업 지시서 작성 형식

사용자가 이 프로젝트에서 "지시서를 써줘"라고 요청하면 다음 규칙을 항상 적용한다.

- 지시서 하나에는 작업 항목을 **정확히 하나만** 담는다. 여러 기능을 한 지시서에 합치지 않는다.
- 여러 작업의 우선순위를 정했더라도 가장 먼저 수행할 항목의 지시서만 작성한다.
- 기존 코드와 관련 문서를 먼저 확인하고, 존재하지 않는 함수·파일·기능을 추측해서 쓰지 않는다.
- 사용자가 제공한 「작업 지시서 — 항목 추가 팝업 붙여넣기 우선 흐름 개편」 예시의 상세도와 구성을 따른다.

지시서는 아래 순서로 작성한다.

1. `📄 작업 지시서 — 작업명` 제목
2. `진행 방식`과 `대상 파일`
3. `📖 작업 배경` — 현재 상태, 문제, 목표 동작, 기존 코드에서 재사용할 부분
4. `🔧 작업 1`, `🔧 작업 2` 형식의 구현 단위
   - 각 작업에 필요한 경우 `먼저 할 일 (추측 금지, 반드시 확인)`을 둔다.
   - 구현 방향과 완료 조건을 구체적으로 적는다.
5. `📖 주의` — 작업 범위, 제외 범위, 회귀 방지 사항
6. `🔧 테스트 확인 사항` — 실제 사용 흐름별 검증 목록
7. `🔧 보고서` — 구현 후 보고할 내용과 MAINTENANCE.md 기준 문서 갱신 대상
8. `🔧 Git` — status/diff 확인, 필요한 파일만 add, commit, push, 배포 확인
   - 작업 완료 보고 시 지시서의 "테스트 확인 사항"을 서술형 요약이 아니라, 목록을 그대로 가져와 항목별 ✅/❌/➖로 표기하도록 명시한다. 누락 항목은 이유를 명시하게 한다.
   - 코드 변경이 있는 모든 커밋은 예외 없이 push 전에 리뷰(자체 리뷰 또는 Codex)를 거치도록 명시한다. 사소해 보이거나 재사용 코드라도 예외 없음. 실수로 먼저 push했다면 사후 리뷰 후 그 사실을 보고서에 명시하게 한다.

추가 작성 원칙:

- 구현자가 추가 질문 없이 작업할 수 있을 정도로 구체적으로 작성한다.
- 기존 함수나 흐름을 재사용하도록 지시하고, 새 라이브러리·구조 변경·무관한 리팩터링은 금지한다.
- 실제 브라우저에서 확인해야 하는 UI 작업은 브라우저 테스트와 회귀 테스트를 명시한다.
- DB/RLS 작업은 위험도를 구분해 지시서에 명시하고, 저위험(SELECT/INSERT/UPDATE)은 MCP로 바로 실행 가능하되 고위험은 반드시 사용자 확인 절차를 거치도록 명시한다.
- 커밋 메시지는 해당 단일 작업만 설명하도록 작성한다.
