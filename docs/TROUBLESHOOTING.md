## 파비콘/앱 아이콘이 최종 확정 디자인과 달랐음

**원인**: 조준경 모티프 파비콘을 처음 적용(`6122fe6`)한 뒤, 디자인 세부사항(삐침 길이, 중앙 레드 점 테두리 두께)이 최종 확정안과 달라 다시 교체해야 했다.

**해결**: 파비콘/앱 아이콘 이미지 파일 세트(`favicon.ico`, `favicon-16/32/192.png`, `apple-touch-icon-180.png`)를 두 차례에 걸쳐 재교체 (`57b2b24`, `440e953`).

**예방**: 파비콘처럼 여러 크기의 이미지가 한 세트로 얽혀 있는 자산은, 디자인을 확정하기 전에 코드에 먼저 반영하지 않고 최종 확정 후 한 번에 전체 세트를 교체하는 편이 반복 커밋을 줄인다.

## 항목이 0개인 태그는 편집모드에서도 섹션 자체가 렌더링되지 않음

**원인**: `renderCards()`의 `groups` 객체가 `filtered.forEach`로만 채워져서, 해당 팀에 아직 항목이 하나도 없는 태그(예: 새 맵의 "위폭")는 `groups`에 키 자체가 생기지 않았다. 그 결과 태그 섹션이 아예 렌더링되지 않고, 편집모드의 "+추가" 타일도 표시되지 않았다.

**해결**: `filtered.forEach` 직후 `tagOrder.forEach(t => { if(!groups[t]) groups[t] = []; })`로 `맵 지명`/`위폭`/`팁` 세 태그를 빈 배열로 미리 채워 넣었다. 단, 일반 사용자 화면에는 영향이 없도록 `sortedTags`를 만들 때 `editMode || groups[t].length > 0` 조건으로 필터링해서, 로그인하지 않은 사용자에게는 기존처럼 항목이 있는 태그만 보이게 했다 (`68238e5`).

**예방**: 화면에 표시할 목록을 "실제 존재하는 데이터"에서만 유도하면, 데이터가 없는 상태의 UI(빈 상태에서의 추가 버튼 등)를 빠뜨리기 쉽다. 태그처럼 값의 종류가 고정된 경우, 데이터 유무와 무관하게 고정 목록(`tagOrder`)을 기준으로 화면 구조를 만드는 편이 이런 누락을 막는다.

## "맵 지명" 태그로 항목 추가 시 유튜브 링크 입력창이 떴음

**원인**: 편집모드 항목 추가 모달(`openAddModal`/`submitItem`)이 처음 이식됐을 때(`62945d9`) 영상(vid) 유형만 지원하도록 만들어졌는데, `isMapLabel` 분기는 `titleWrap`(제목/설명)만 숨기고 유튜브 링크 입력(`mVideoUrl`)은 모든 태그에서 항상 노출하도록 되어 있었다. 즉 "유형을 잘못 전환하는" 버그가 아니라, 애초에 이미지 유형 자체와 유형별 표시 분기가 코드에 없었다(레거시 Admin의 `type-toggle`/이미지 업로드가 User 사이트로 이식되지 않은 상태, `docs/architecture/admin-flow.md` 참고).

**해결**: 모달에 `videoWrap`/`imageWrap` 두 컨테이너를 추가하고, `openAddModal()`에서 `isMapLabel`이면 `videoWrap`을 숨기고 `imageWrap`(파일 선택 + Ctrl+V 붙여넣기)만 보이도록 했다. `submitItem()`도 `isMapLabel`이면 이미지 파일을 Storage `media` 버킷의 `items/{timestamp}.jpg`에 업로드해 `img_url`로 저장하는 분기를 별도로 추가했다. "위폭"/"팁" 태그는 기존처럼 영상 입력만 노출된다.

**예방**: 특정 태그에서만 입력 항목을 숨기는 조건 분기를 추가할 때는, 그 태그와 무관해 보이는 다른 입력 필드도 실제로 조건에 맞게 숨겨지는지 함께 확인한다. 일부 필드만 조건부로 처리하고 나머지를 빠뜨리면 "숨겨야 할 UI가 남아있는" 형태의 버그가 생기기 쉽다.

## 클립 구간 끝(end) 슬라이더가 뻑뻑하게 움직였음

**원인**: 시작/끝 슬라이더의 "서로 교차 금지" 로직(`onClipStartInput`/`onClipEndInput`)이, 반대쪽 슬라이더의 `.value`를 직접 재할당해서 경계를 강제하는 방식이었다. 두 핸들이 서로 반대 방향으로 겹쳐서 지나가는 드래그(예: 끝 핸들을 시작 핸들 쪽으로 계속 내리는 경우)에서는, 교차가 시작된 시점부터 드래그가 끝날 때까지 **매 `oninput` 틱마다** 반대쪽 슬라이더의 `.value`를 계속 다시 써야 했다 — 끝 슬라이더의 기본 위치가 항상 맨 오른쪽(전체 길이)이라 사용자가 끝 핸들을 왼쪽으로 크게 끌 때 이 "교차 유지 갱신"에 걸리는 구간이 시작 핸들보다 훨씬 넓어서, 유독 끝 쪽이 뻑뻑하게 느껴졌다.

**해결**: 반대쪽 슬라이더의 값을 직접 미는 대신, 각 슬라이더 자신의 네이티브 `min`/`max` 속성으로 서로 넘나들 수 없는 배타적 범위를 걸었다(`startRange.max = clipEnd - 1`, `endRange.min = clipStart + 1`). 이러면 브라우저가 각 슬라이더를 자기 범위 안에서만 자연스럽게 멈추게 해주므로, 드래그 중에 반대쪽 슬라이더의 값을 계속 다시 쓸 필요가 없어진다. 같은 작업에서 위/아래로 분리돼 있던 슬라이더 2개를 겹친 단일 트랙 구간 바로 바꾸면서 이 방식으로 재구현했다.

**예방**: 두 개의 값이 서로 침범하지 못하게 막아야 하는 UI(구간 선택, 최소/최대 필터 등)를 만들 때는, "상대방 값을 직접 밀어내는" 방식보다 "각자의 허용 범위(min/max)를 좁혀서 네이티브 컨트롤이 알아서 멈추게 하는" 방식을 먼저 고려한다. 전자는 교차 구간에서 매 이벤트마다 상대방 상태를 다시 쓰게 되어 성능/체감 반응성 문제가 생기기 쉽다.

## 클립 구간 채움 바가 끝 손잡이보다 오른쪽으로 튀어나와 보였음

**진단(데이터 vs 렌더링 구분)**: 저장되는 `clipEnd`(→ `submitItem()`의 `clip_end`), 화면 라벨/끝 슬라이더의 `.value`, 재생 시 구간 반복에 쓰이는 `openOverlay()`의 `end`(→ `it.clip_end`) 세 지점을 코드로 추적한 결과, 셋 다 **동일한 하나의 값**을 그대로 참조한다 — `onClipEndInput()`이 `clipEnd = Number(endRange.value)`로 설정 → `submitItem()`이 그 `clipEnd`를 그대로 `insert` → Supabase에 저장된 값을 그대로 `it.clip_end`로 다시 읽어 재생에 쓴다. 값이 갈라지는 지점은 없었다(헤드리스 테스트로 세 지점이 같은 숫자를 갖는지 직접 비교해 확인). 즉 **데이터 문제가 아니라 순수 렌더링(CSS) 문제**였다.

**렌더링 원인**: `updateClipRangeFill()`이 채움 바의 `left`/`width`를 `value / duration * 100%`로, 즉 슬라이더 값이 트랙의 0~100% 전체 폭에 선형으로 대응한다고 가정해 계산했다. 하지만 네이티브 `<input type=range>`의 손잡이(thumb)는 트랙 전체 폭이 아니라 `트랙폭 - 손잡이폭`만큼만 움직이고, 손잡이 중심은 항상 손잡이 반지름(이 프로젝트의 thumb는 16px → 반지름 8px)만큼 안쪽에서 시작·종료한다. 그래서 값이 `duration`(최대)에 가까울수록 실제 손잡이 위치보다 채움 바 오른쪽 끝이 더 오른쪽으로("반지름만큼") 튀어나와 보였다.

**해결**: 채움 바 위치 계산을 손잡이 반지름을 보정한 `calc()` 식으로 바꿨다 — `left: calc(8px + (100% - 16px) * startFraction)`, `width: calc((100% - 16px) * (endFraction - startFraction))`. 이는 브라우저가 실제로 손잡이를 배치하는 공식과 동일해서, 채움 바 양끝이 항상 두 손잡이의 정확한 중심과 일치한다.

**예방**: 네이티브 `<input type=range>` 위에 커스텀 진행바/채움 바를 겹쳐 그릴 때는 값의 비율을 트랙의 퍼센트에 그대로 매핑하면 안 된다 — 손잡이 크기만큼 항상 보정(`calc(반지름 + (100% - 손잡이폭) * 비율)`)해야 실제 손잡이 위치와 일치한다. 이 프로젝트처럼 손잡이 폭을 CSS에서 고정값(16px)으로 정했다면, 채움 바 계산에도 같은 상수를 그대로 반영해야 한다.

## 쇼츠(짧은 영상)에서 클립 슬라이더 트랙이 압축되어 보이고, 끝 손잡이가 뻑뻑하게 느껴졌음

**원인**: `loadClipPlayer()`가 `onReady` 이벤트 안에서 `clipDuration = Math.floor(e.target.getDuration()) || 0`을 **단 한 번만** 호출해 그 결과를 곧바로 슬라이더 `min`/`max`에 반영했다. 그런데 유튜브 IFrame API는 `onReady` 시점에 `getDuration()`이 최종적으로 정확한 값을 돌려준다고 보장하지 않는다 — 특히 재생을 시작하지 않은 채(이 플레이어는 `autoplay` 없이 cue만 되어 있음) 재생 시간이 매우 짧은 쇼츠는 메타데이터가 늦게 채워져서, `onReady` 직후 `getDuration()`이 0이거나 실제(예: 31초)보다 훨씬 작은 값(예: 1~2초)을 돌려주는 경우가 있었다. 이 잘못된 작은 값이 `clipDuration`으로 그대로 확정되면서:
- 슬라이더의 `max`가 실제 영상 길이보다 훨씬 작게 설정돼 트랙 전체가 눌린 것처럼(압축되어) 보였다.
- 가용 범위 자체가 몇 초에 불과하다 보니, `step="1"` 단위로 조금만 움직여도 전체 구간에서 차지하는 비중이 커서 "1씩 거꾸로 튀는" 것처럼 뻑뻑하게 느껴졌다(이전에 고쳤던 "끝쪽이 뻑뻑하다" 버그와는 원인이 다르다 — 그건 상대 슬라이더 값을 서로 밀어내던 로직 문제였고, 이번 건은 애초에 duration 자체가 잘못 확정된 문제다).

풀영상(예: 23분)에서 이 문제가 눈에 띄지 않았던 이유는, `onReady` 시점에 이미 정확한 duration이 채워져 있는 경우가 많았기 때문으로 보인다 — 다만 이는 유튜브 쪽 타이밍에 달려 있어 보장된 동작은 아니었다.

**해결**: `onReady`에서 `getDuration()`을 한 번 믿고 확정하는 대신, `applyClipDuration()`이라는 별도 함수로 "duration 확정 시 슬라이더에 반영"하는 로직을 분리하고, `onReady` 안에서는 150ms 간격으로 `getDuration()`을 반복 호출해 **두 번 연속 같은 양수 값**이 나올 때만 `applyClipDuration()`을 호출하도록 바꿨다(최대 20회/약 3초 시도 후에는 그 시점의 값으로 폴백 확정). 또한 `openAddModal()`과 `loadClipPlayer()` 시작 시점에 `applyClipDuration(0)`을 호출해, 새 영상을 불러오는 동안 이전 영상의 duration/슬라이더 범위가 화면에 남아있지 않도록 했다.

**테스트**: jsdom으로 `YT.Player`를 모킹해 ①풀영상처럼 처음부터 안정된 값이 나오는 경우, ②쇼츠처럼 `0 → 1 → 31 → 31`로 값이 바뀌다 안정되는 경우, ③처음부터 정확한 짧은 영상, ④끝까지 값이 계속 바뀌어 폴백(20회)까지 가는 경우 네 가지를 확인 — 모두 최종적으로 올바른(혹은 폴백) duration이 슬라이더에 반영됨을 검증했다. 실제 유튜브 서버에서 쇼츠의 duration이 정확히 몇 번째 호출에서 안정되는지는 브라우저 환경에서만 확인 가능하다.

**예방**: 유튜브 IFrame API처럼 "이벤트가 발생했다고 해서 관련 메타데이터(duration, 화질 목록 등)가 항상 최종값이라고 보장되지 않는" 외부 API를 다룰 때는, 이벤트 콜백 안에서 값을 한 번만 읽고 확정하지 말고 "같은 값이 연속으로 나오는지" 같은 안정성 확인 후 확정하는 편이 안전하다. 특히 이 문제는 길이가 긴 콘텐츠보다 짧은 콘텐츠(쇼츠 등 메타데이터 로딩이 상대적으로 늦게 완료되는 경우)에서 더 잘 드러나므로, 다양한 길이로 테스트해야 발견하기 쉽다.

## 클립 구간 채움 바가 (반지름 보정, duration 폴링 확정 이후에도) 여전히 손잡이 밖으로 튀어나왔음

**지난 수정이 불완전했던 이유**: 위 "채움 바가 끝 손잡이보다 오른쪽으로 튀어나와 보였음" 항목에서 손잡이 반지름(8px)을 보정한 `calc(8px + (100% - 16px) * (value/clipDuration))` 공식으로 고쳤지만, 이 공식은 **"각 슬라이더의 실제 min~max 폭이 항상 `[0, clipDuration]`"이라는 잘못된 전제** 위에 있었다. 그런데 "끝쪽이 뻑뻑함" 항목에서 채택한 교차 방지 방식은 정확히 그 전제를 깨고 있었다 — `onClipEndInput`이 끝을 움직일 때마다 `startRange.max = clipEnd - 1`을, `onClipStartInput`이 시작을 움직일 때마다 `endRange.min = clipStart + 1`을 계속 재할당해서, 두 슬라이더의 실제 `min~max` 폭을 `clipDuration`보다 계속 좁혀놓았다. 브라우저는 손잡이를 `value/clipDuration`이 아니라 `(value - 자신의 min) / (자신의 max - 자신의 min)` 비율로 그리기 때문에, 이 폭이 좁아질수록 손잡이의 실제 화면 위치와 채움 바 계산이 서로 어긋났다. 즉 duration 캐싱이나 호출 누락 문제가 아니라(둘 다 재확인했지만 문제 없었음 — `updateClipRangeFill()`은 `oninput`마다, `applyClipDuration()` 안에서 duration 갱신 시마다 빠짐없이 호출되고 있었다), **손잡이 반지름 보정 공식 자체가 "분모"로 잘못된 값(clipDuration)을 계속 썼던 것**이 두 차례 수정에도 버그가 재현된 진짜 원인이다.

Playwright(Chromium)로 duration=35, 시작=9, 끝=19인 상태를 실제로 재현해 손잡이가 렌더링되는 실제 픽셀 위치를 확인한 결과, 이 어긋남은 단순히 채움 바만의 문제가 아니라 **"시작" 손잡이가 화면상 "끝" 손잡이보다 오른쪽에 그려지는(두 손잡이 자체가 뒤바뀌어 보이는)** 더 근본적인 현상까지 만들어내고 있었다(시작 슬라이더의 실제 폭은 `[0, 18]`이라 9는 50% 지점, 끝 슬라이더의 실제 폭은 `[10, 35]`라 19는 36% 지점 — 값은 9<19로 정상이지만 화면 위치는 50%>36%로 역전). 채움 바가 "손잡이 사이보다 훨씬 길게" 그려진 것은 이 역전 때문이었다.

**해결**:
1. `updateClipRangeFill()`이 `value/clipDuration` 대신 각 슬라이더 **자기 자신의 살아있는 `min`/`max` 속성**을 읽어 `(value - min)/(max - min)`으로 비율을 구하도록 바꿨다. 두 손잡이가 어떤 이유로든 서로 다른 min/max를 갖게 되더라도, 채움 바는 항상 실제 렌더링된 손잡이 위치 사이에만 그려진다(`left = min(시작비율, 끝비율)`, `right = max(시작비율, 끝비율)`로 계산해 순서가 뒤바뀌어도 안전).
2. 근본 원인인 "상대 슬라이더의 min/max를 좁히는" 교차 방지 방식 자체를 제거했다. 이제 `applyClipDuration()`/`syncClipSliders()`는 두 슬라이더의 `min`/`max`를 항상 `[0, clipDuration]`으로 고정해서 유지하고, 교차 방지는 `onClipStartInput`/`onClipEndInput`에서 **자기 자신의 value만** `Math.max(e-1, 0)` / `Math.min(s+1, clipDuration)`으로 clamp하는 방식으로 바꿨다. 상대방의 값이나 min/max는 절대 건드리지 않으므로, "끝쪽이 뻑뻑함" 버그(상대 슬라이더 값을 매 틱마다 재할당해서 생긴 문제)를 다시 만들지 않으면서도, 손잡이의 실제 화면 위치가 항상 `value/clipDuration`과 정확히 일치하게 됐다.

**검증(실제 브라우저)**: jsdom이 아니라 Playwright로 Chromium을 직접 띄워서, index.html의 CSS(`.clip-range-*`)와 수정된 JS 함수를 그대로 복사한 재현 페이지를 만들어 확인했다(index.html 자체는 Supabase 로그인 + 편집모드 + 유튜브 IFrame API에 의존하고 있어 앱 전체를 그대로 구동하며 값을 세팅하기 어려워, 동일한 마크업/CSS/함수를 쓰는 축소 재현 페이지로 검증하는 방식을 택했다). 슬라이더 컨테이너의 `getBoundingClientRect()`를 기준으로, 채움 바의 실제 좌/우 픽셀과 "손잡이가 렌더링되어야 할 위치"(동일한 `calc(8px + (100%-16px)*비율)` 공식을 손잡이 쪽에도 그대로 적용해 노란 세로선으로 겹쳐 그림, 실제 화면 스크린샷으로 원 중심과 정확히 겹치는지 육안 확인)를 비교했다.

| 시나리오 | duration | 시작 | 끝 | 채움 바 left(px) | 채움 바 right(px) | 손잡이 위치(left/right, px) | 일치 여부 |
|---|---|---|---|---|---|---|---|
| 스크린샷1 재현 (0:09~0:27) | 35 | 9 | 27 | 132.45 | 381.36 | 132.45 / 381.36 | 일치 |
| 스크린샷2 재현 (0:09~0:19, 좁힌 구간) | 35 | 9 | 19 | 132.45 | 270.73 | 132.45 / 270.73 | 일치 |
| 넓은 영상에서 같은 구간 (0:09~0:27) | 60 | 9 | 27 | 80.59 | 225.78 | 80.59 / 225.80 | 일치(반올림 오차 0.02px) |
| 쇼츠급 짧은 영상, 좁은 구간 (0:09~0:19) | 20 | 9 | 19 | 225.80 | 467.80 | 225.80 / 467.80 | 일치 |

수정 전 코드로 같은 표를 뽑으면 "스크린샷2 재현" 행에서 채움 바가 132px~271px(139px 폭)로 계산되는데, 실제 손잡이는 182px~250px(68px 폭)에 그려져 있어 채움 바가 손잡이 바깥 양쪽으로 크게 튀어나오고, 심지어 "시작" 손잡이(250px)가 "끝" 손잡이(182px)보다 오른쪽에 그려지는 역전까지 확인됐다 — 수정 후에는 모든 시나리오에서 채움 바 양 끝이 손잡이 위치와 소수점 둘째 자리까지 일치한다.

**예방**: 손잡이 크기를 보정한 `calc(반지름 + (100% - 손잡이폭) * 비율)` 공식 자체는 맞더라도, 그 "비율"을 구하는 분모가 실제로 그 슬라이더에 적용된 `min`/`max`와 같은지 항상 재확인해야 한다. 두 개의 `<input type=range>`가 서로의 값으로 상대방 min/max를 동적으로 좁히는 방식(교차 방지 등)을 쓰고 있다면, 커스텀 진행바 계산은 "고정된 전체 값" 대신 반드시 그 슬라이더의 **그 순간의 살아있는 min/max**를 읽어서 비율을 구해야 한다. 더 근본적으로는, 두 손잡이의 min/max를 서로 좁히는 방식 자체가 "손잡이의 화면 위치가 실제 값에 비례하지 않게 되는" 부작용을 만들 수 있으므로, 교차 방지는 (min/max를 좁히기보다는) 각 입력 핸들러에서 자기 자신의 value만 clamp하는 방식을 우선 고려한다.

## 편집모드 영상 미리보기에서 지정 구간 밖 재생이 가능했음

**원인**: `markClipStart()`/`markClipEnd()`와 슬라이더 핸들러는 `clipStart`/`clipEnd` 값을 저장하고 화면만 동기화했다. 실제 재생 위치를 감시하는 로직은 일반 카드 오버레이의 `ytPlayer`/`clipTimer`에만 있었고, 편집 미리보기의 별도 플레이어 `clipYtPlayer`에는 없었다. 따라서 관리자가 YouTube 컨트롤바로 선택 구간 밖을 탐색해도 되돌리는 주체가 없었다.

**해결**: 편집 전용 `clipPreviewTimer`를 추가해 양쪽 경계가 모두 지정된 동안 250ms 간격으로 현재 위치를 확인한다. 허용 범위는 `clipStart <= current < clipEnd`이며, 범위 밖이면 `clipStart`로 이동한다. 감지 당시 `PLAYING`/`BUFFERING`/`ENDED` 상태는 계속 재생하고 `PAUSED` 상태는 이동 후 다시 일시정지한다. `clearClip()`, 새 영상 로드, 이미지 흐름 전환, 모달 열기/닫기에서 타이머를 정리해 중복과 죽은 플레이어 접근을 막았다.

## "구간 끝 지정" 버튼을 누르면 곧바로 시작 지점으로 되돌아갔음

**원인**: 위 항목에서 추가한 폴링이 `current >= clipEnd`(등호 포함, 여유 없음)를 이탈로 판정하는데, `markClipEnd()`는 `clipEnd = Math.floor(getCurrentTime())`로 저장한다. `Math.floor`는 항상 실제 값 이하이므로, 저장 직후의 실제 재생 위치는 이미 `clipEnd` 이상이다 — 즉 "구간 끝 지정" 버튼을 누르는 순간 그 자체가 첫 폴링(최대 250ms 후)에서 "구간 밖"으로 오판되어 `clipStart`로 튕겨 나갔다.

**해결**: `markClipEnd()`가 `clipEndMarkGraceUntil = Date.now() + 1300`(절삭 오차 1초 미만 + 폴링 대기 250ms를 덮는 여유)을 함께 저장하고, 폴링은 이 유예 시간 안에서만 상한 검사를 건너뛴다. 처음에는 상한 검사 자체에 전역으로 `+1`초 여유를 주는 방식으로 고쳤으나(Codex 리뷰에서 지적), 이 방식은 슬라이더로 정확히 지정한 `clipEnd`까지 항상 느슨해지고 최악 지연(약 1.25초)도 완전히 못 덮는 문제가 있어, 마킹 시점에만 짧게 켜지는 유예로 교체했다. 슬라이더 입력(`onClipStartInput`/`onClipEndInput`)과 구간 초기화(`clearClip`/`loadClipPlayer`/`openAddModal`)에서는 이 유예를 즉시 0으로 되돌려, 슬라이더로 새로 지정한 값은 항상 정밀한 `current >= clipEnd` 검사를 그대로 받는다.

**예방**: "현재 재생 위치를 반올림/절삭해 저장한 값"을 곧바로 "그 값과 실제 재생 위치를 비교하는 조건"의 경계로 재사용하면, 반올림 방향에 따라 저장 직후 조건이 항상(또는 거의 항상) 참이 되는 자기유발 버그가 생기기 쉽다. 이런 조건에는 방향과 지속시간을 명확히 좁힌 예외(유예 시간/플래그)를 두고, 그 예외가 다른 입력 경로(슬라이더 등)의 정밀도까지 갉아먹지 않는지 확인한다.

**예방**: 서로 다른 YouTube 플레이어 인스턴스는 감시 타이머도 분리하고, 플레이어를 `destroy()`하는 모든 경로에서 해당 타이머를 먼저 정리한다. 구간 제한은 한쪽 경계만 있을 때 활성화하지 않는다.

## 편집 모달에서 재생 상태 전환 후 클립 종료 감시가 복구되지 않을 수 있었음

**증상**: 시작·끝을 모두 지정한 편집 미리보기에서 YouTube 재생 버튼을 눌렀을 때, 감시 타이머가 없는 상태라면 `clipEnd`를 지나도 `clipStart`로 돌아오지 않았다.

**실제 원인**: 구간 감시는 `markClipStart()`/`markClipEnd()`와 슬라이더 입력 때만 `syncClipPreviewTimer()`로 연결됐다. 모달 종료·영상 교체·이미지 전환에서는 의도적으로 정리됐지만, 플레이어 자체의 재생·일시정지·버퍼링·종료 상태 변화에는 감시를 다시 연결하는 경로가 없었다. 일반 카드 오버레이는 `onStateChange(PLAYING)`에서 `clipTimer`를 연결한다는 차이가 있었다.

**수정**: 편집 플레이어의 `onStateChange`에서 `clipPreviewTimer`가 없을 때만 기존 `syncClipPreviewTimer()`를 호출한다. 기존 타이머가 있으면 건드리지 않아 중복 타이머를 만들지 않고, 구간이 한쪽만 지정됐거나 초기화된 경우에는 `syncClipPreviewTimer()`의 기존 가드가 그대로 종료한다. `clipEndMarkGraceUntil`의 1.3초 유예와 `[clipStart, clipEnd)` 경계 판정은 변경하지 않았다.

**재현 테스트**: 실제 Chrome 150과 YouTube iframe에서 5~9초 구간을 버튼으로 지정하고 iframe 재생 버튼을 눌러 반복 재생됨을 확인했다. 일시정지 상태에서 12초로 이동하면 5초로 복귀하면서 일시정지를 유지했다. 추가 Node 자가검사는 타이머가 없는 상태의 상한 이탈이 시작점 seek와 일시정지 유지를 수행하고, 플레이어 상태 콜백에 감시 복구 호출이 남아 있는지 확인한다.

## 그룹 D 1단계 클릭 추적: "호출 누락"으로 잘못 진단했으나 실제로는 미커밋·미배포 상태였음

**증상**: `trackClick()` 구현을 완료했다고 보고했지만, 운영 Vercel URL에서 `console.log(openOverlay.toString())`으로 확인했을 때 `trackClick` 관련 코드가 전혀 보이지 않았다.

**잘못된 최초 진단**: 배포본만 보고 "`openOverlay()`에 `trackClick()` 호출이 누락된 버그"로 판단했다. 실제로는 로컬 `index.html`을 확인하지 않고 배포본 증상만으로 원인을 추측한 것이었다.

**실제 원인**: `git status`로 로컬을 확인한 결과 `trackClick()` 정의와 `void trackClick(id)` 호출이 이미 `index.html`에 존재했지만 커밋되지 않은 상태(`Changes not staged for commit`)였다. `git log --all -S"trackClick"`로도 관련 커밋이 전혀 없었고 `origin/main`에도 반영되지 않았다 — 즉 구현 자체는 끝났으나 커밋·푸시가 되지 않아 Vercel에 배포되지 않은 것이 정확한 원인이었다. 배포본에서 문자열 자체가 안 보였던 증상과도 정확히 들어맞는다.

**수정**: 별도 코드 작성 없이 기존 미커밋 diff를 원 지시서 요구사항(에러/예외 둘 다 캐치, `alert` 금지, `openOverlay()` 최상단 fire-and-forget 호출)과 대조해 일치함을 확인한 뒤 그대로 커밋(`b743418`)하고 푸시했다. 배포 후 운영 URL의 raw HTML과 Playwright로 실제 카드 클릭 시 `POST .../item_clicks` 요청을 재확인했다.

**예방**: 코드 수정을 완료했다고 보고하기 전에 반드시 `git status`로 미커밋 여부를 확인한다. 배포된 정적 사이트의 동작을 검증할 때는 로컬 파일 상태를 먼저 확인하지 않고 배포본 증상만으로 원인을 추측하지 않는다 — "코드가 잘못됐다"와 "맞는 코드가 아직 배포되지 않았다"는 증상이 같아 보여도 진단·수정 방법이 전혀 다르다.
