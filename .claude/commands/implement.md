# /implement <지시서 파일 경로>

지시서 하나를 받아 설계 리뷰 → 구현 → 커밋 전 코드 리뷰 → 커밋·푸시까지 사람 개입을 최소화해서 끝까지 진행하는 파이프라인이다.

```
지시서와 실제 코드 확인
  ↓
Codex CLI 읽기 전용 설계 리뷰
  ↓
명백한 사실 오류 · 안전한 보완점 자동 반영 (그 외엔 분류에 따라 기록/확인)
  ↓
Claude Code 구현 · 테스트 · 문서 갱신
  ↓
Codex CLI 읽기 전용 커밋 전 코드 리뷰
  ↓
지적된 명백한 결함 자동 수정 (최대 2회 반복)
  ↓
통과하면 git add → commit → push → 배포 확인
  ↓
최종 보고
```

## 핵심 아키텍처 결정

- **`codex:codex-rescue` 서브에이전트는 이 파이프라인에서 쓰지 않는다.** 그 서브에이전트는 "Codex 결과를 사용자에게 그대로 반환하고 턴을 끝낸다"는 게 설계 목적이라, 자동으로 다음 단계(분류→구현→재리뷰)로 이어갈 수 없다.
- 대신 **공개 Codex CLI(`codex exec`)를 Bash 도구로 직접 호출**한다(Agent/Task 도구를 거치지 않는다). 읽기 전용으로만 호출하며, 설계 리뷰와 커밋 전 코드 리뷰 둘 다 이 방식을 쓴다.

## 입력 계약

- `$ARGUMENTS`는 지시서 파일 경로 정확히 하나여야 한다.
- 판정 순서: 먼저 `$ARGUMENTS` 전체(양끝 공백만 제거)를 **하나의 경로**로 보고 파일이 존재하는지 확인한다(공백이 포함된 경로도 여기서는 그대로 인식된다). 존재하면 그 파일을 지시서로 쓴다.
- 그 경로가 존재하지 않을 때만, 공백 기준으로 나눠 토큰이 2개 이상인지 본다 — 2개 이상이면 "복수 파일 지정"으로 간주한다(공백이 포함된 단일 경로는 지원하지 않는다는 뜻이므로, 보고 시 공백 없는 경로로 다시 시도해달라고 안내한다).
- 다음 중 하나라도 해당하면 구현을 시작하지 않고 무엇이 문제였는지 + 올바른 사용법(`/implement <지시서 경로>`)을 보고한 뒤 중단한다:
  - `$ARGUMENTS`가 비어 있음
  - 위 판정으로 복수 파일임
  - 단일 경로로도, 공백 분리 토큰 어느 쪽으로도 실제 파일을 찾지 못함

## 사람이 개입하는 5가지 지점

이 파이프라인이 절대 축소·우회하지 않는 지점. 분류가 애매하면 항상 이쪽으로 기운다.

1. 터미널/VS Code 도구 실행 권한 확인(`Yes` 클릭) — 평소와 동일, Claude Code 기본 동작이라 이 명령이 따로 처리하지 않는다
2. `docs/CLAUDE_CODE_RULES.md`의 "SQL 실행 규칙" 고위험 작업(DELETE/DROP/TRUNCATE/CREATE TABLE/ALTER TABLE/RLS 정책 생성·변경·삭제) — 실행 전 무엇을 하려는지와 되돌릴 수 없다는 점을 명시하고 "진행할까요?" 확인
3. 비용이 발생하거나, 인증·권한을 완화하거나, 데이터 손실 가능성이 있는 작업 — 그 시점에 멈추고 확인
4. Codex와 Claude 둘 다 정답을 확정하지 못한 핵심 UX/설계 선택 — AskUserQuestion으로 확인, 답변 전에는 그 부분 구현을 진행하지 않는다
5. 4단계에서 최초 리뷰 후 수정·재리뷰를 최대 2회까지 반복해도 통과하지 못한 경우 — 커밋하지 않고 중단, 남은 지적 사항을 그대로 보고

## codex exec 호출 방식

**하드코딩 금지.** 아래는 2026-07-22, `codex-cli 0.144.6` 기준으로 `codex exec --help`/`codex exec review --help`로 직접 확인한 옵션이다. 이 명령을 실행할 때마다 먼저 두 `--help`를 다시 실행해서 옵션이 그대로인지 확인하고, 달라졌으면 그에 맞게 호출을 조정한다.

- **호출은 항상 Bash 도구로 한다(PowerShell 도구 아님).** Bash 도구에서 `codex exec --help`는 항상 정상 종료 코드(0)로 끝나는 것을 확인했다(2026-07-22). PowerShell 도구에서는 같은 호출이 실행 환경에 따라 다르게 관찰됐다 — 어떤 세션에서는 `codex`/`codex.ps1` 실행 자체가 막히고, 다른 세션에서는 `codex`/`codex.cmd` 모두 출력은 정상인데 종료 코드만 `-1`/`255`로 잘못 보고됐다(정확한 원인은 세션마다 달라 특정하지 않는다 — ExecutionPolicy로 단정하지 않는다). 이런 불안정성 때문에 정상 호출을 실패로 오판할 수 있으니, codex exec 호출은 전부 Bash 도구를 쓴다. 아래 예시도 전부 Bash(POSIX sh) 문법이다.
- **읽기 전용 실행**은 `-s read-only`(sandbox 모드)로 강제한다. 파일 수정이 불가능하고 셸 명령도 읽기 전용으로만 실행된다.
- **응답만 따로 저장**하려면 `-o <파일>`(`--output-last-message`)을 쓴다. 터미널 진행 로그와 최종 응답이 섞여 파싱이 깨지는 것을 막기 위함이다. 이 출력 파일은 **저장소 안이 아니라 세션 스크래치 디렉터리**(시스템 프롬프트의 Scratchpad Directory 안내를 따름)에 쓴다.
- **긴 프롬프트는 CLI 인자로 직접 욱여넣지 않는다.** 따옴표·개행·백틱이 섞이면 셸 이스케이프가 깨지기 쉽다. 대신 프롬프트를 스크래치 디렉터리의 임시 파일에 먼저 쓰고, `codex exec ... -` (인자로 `-`)로 stdin에서 읽게 한다.
- 두 리뷰 호출 모두 Bash 도구의 `timeout` 파라미터를 여유 있게(예: 300000~600000ms) 설정한다.

**설계 리뷰(1단계, 아직 diff 없음)** — 프롬프트 파일 + stdin 방식:
```bash
codex exec -s read-only -C "<레포 루트 절대경로>" -o "<스크래치>/design-review.txt" - < "<스크래치>/design-review-prompt.txt"
```

**커밋 전 코드 리뷰(4단계, diff 있음)** — 전용 서브커맨드 사용:
```bash
codex exec review --uncommitted -c sandbox_mode="read-only" -o "<스크래치>/code-review.txt"
```
`review` 서브커맨드에는 `-s/--sandbox` 숏핸드가 없다(2026-07-22 기준 `--help`로 확인). 대신 범용 설정 오버라이드 `-c sandbox_mode="read-only"`를 붙이면 실제로 읽기 전용으로 강제된다 — 직접 실행해서 실행 헤더의 `sandbox: read-only` 표시와 정상 종료(exit 0), `-o` 파일 정상 생성까지 확인했다(2026-07-22, codex-cli 0.144.6). 이 플래그 없이 실행하면 `workspace-write`가 기본값이므로 반드시 붙인다. `--uncommitted`가 staged/unstaged/untracked 변경사항을 대상으로 하므로 별도 프롬프트 조립이 필요 없다.
**주의**: 이 호출을 `| head` 등으로 파이프해서 중간에 자르지 않는다 — 출력 스트림이 끊기면 codex 프로세스가 비정상 종료 코드(예: 101)를 낼 수 있다(2026-07-22 실측, 실제 회귀는 아니었음). `-o` 파일로 결과를 받고 필요하면 그 파일을 나중에 Read한다.

**실패 처리(추측 금지, 즉시 중단·보고)**: 아래 중 하나라도 해당하면 그 시점에 멈추고, 실행한 명령·종료 코드·stderr를 그대로 보고한다.
- Bash 도구 타임아웃 발생
- 종료 코드가 0이 아님
- `-o` 출력 파일이 없거나 비어 있음
- 출력이 명백한 오류/거부 메시지뿐이고 리뷰 내용이라 볼 수 없음(형식 불일치)

## 1단계 — 설계 리뷰

1. 지시서(`$ARGUMENTS`) 전체를 읽는다.
2. 지시서가 언급하는 실제 코드·파일·함수를 grep/read로 직접 확인한다(Codex 응답을 무비판적으로 받아들이지 않기 위한 교차검증 기반).
3. 위 "설계 리뷰" 호출 방식으로 Codex에게 요청한다. 프롬프트에는 최소한 다음을 포함한다:
   - 아직 구현 전이며, 코드/파일을 수정하지 말고 리뷰만 하라는 지시
   - 지시서 절대경로(Codex가 read-only 권한으로 직접 읽게 함) — 또는 지시서 전체 내용
   - "지시서가 언급하는 함수명·파일 경로·현재 동작이 실제 저장소와 일치하는지 확인하라"
   - "발견한 항목을 번호로 나열하고, 각각을 '사실 오류/코드와 다른 가정'과 '설계상 애매한 지점'으로 구분해서 표시하라. 문제가 없으면 그렇게만 답하라."
4. 실패 시 위 "실패 처리"에 따라 중단.

## 2단계 — 리뷰 결과 분류 및 처리

Codex가 지적한 항목과 Claude가 직접 확인한 내용을 합쳐 아래로 분류한다.

| 분류 | 조건 | 처리 |
|---|---|---|
| 자동 수정 | 사실 오류, 실제 코드와 안 맞는 가정, 명백한 계산/논리 실수 | 지시서의 의도를 지키며 구현에 바로 반영(지시서 파일 자체는 고치지 않는다) |
| 자동 결정 | Codex가 "애매하다"고 했지만 지시서를 다시 읽으면 이미 답이 있음 | 지시서의 어느 부분이 근거인지 남기고 그대로 진행 |
| `docs/KNOWN_ISSUES.md` 기록 후 스킵 | 이번 작업 범위 밖의 기존 결함을 우연히 발견 | 기존 문서 형식 그대로 추가, 이번 구현에서는 고치지 않는다 |
| `docs/TODO.md` 기록 | 지금 판단이 불필요한 단순 후속 개선 아이디어 | "아이디어" 섹션에 추가 |
| 사용자에게 실시간 확인 | 핵심 UX/설계 선택(Codex·Claude 둘 다 정답 미확정) 또는 위 "5가지 지점"의 2·3번에 해당 | AskUserQuestion으로 그 시점에 확인, 답변 전에는 그 부분을 진행하지 않는다 |

분류가 애매하면 "사용자에게 확인"으로 기운다.

## 3단계 — 구현

- 2단계 분류에 따라 구현한다.
- 테스트·문서 갱신은 `docs/CLAUDE_CODE_RULES.md`/`docs/PROMPTS.md`/`docs/MAINTENANCE.md`의 기존 규칙을 그대로 따른다 — 이 명령이 별도 규칙을 새로 만들지 않는다.

## 4단계 — 커밋 전 코드 리뷰 (최대 2회 반복)

1. `git status`/`git diff`로 실제 변경사항(staged+unstaged+untracked)을 확인한다.
2. 위 "커밋 전 코드 리뷰" 호출 방식으로 `codex exec review --uncommitted`를 실행한다.
3. 결과 확인:
   - 지적 사항 없음 → 5단계로 진행
   - 명백한 결함이 지적됨 → 수정 후 다시 2번부터 반복
4. 이 "수정 → 재리뷰" 사이클은 **최대 2회**까지만 허용한다(최초 리뷰 1회 + 재리뷰 최대 2회 = 총 최대 3회 리뷰 호출).
5. 2회 반복 후에도 통과하지 못하면: 자동 수정을 멈추고, 커밋하지 않고, 남은 지적 사항을 사용자에게 그대로 보고한 뒤 중단한다(사람 개입 지점 5번).

## 5단계 — 커밋·푸시·배포 확인

리뷰를 통과했으면(최초 통과 또는 4단계 2회 이내 통과):

1. `git status`/`git diff` 확인(`.claude/settings.local.json` 제외)
2. 필요한 파일만 `git add`
3. `git commit`(`docs/PROMPTS.md`의 커밋 메시지 규칙에 따름)
4. `git push`
5. Vercel 배포 확인

## 알려진 한계

`/implement`가 4단계에서 자체적으로 코드 리뷰를 수행한 뒤에도, 턴이 끝나는 시점에 기존 `stop-review-gate`가 별도로 한 번 더 자동 발동할 수 있다(현재 Stop 훅이 실행 경로를 구분하지 못해 `/implement`를 거쳤는지 모른다). 이는 중복 리뷰지만 안전 방향의 중복이라 허용한다 — 추가로 뭔가 지적되면 그 시점에 다시 대응한다. `stop-review-gate` 자체는 `/implement`를 거치지 않는 일반 코드 작업의 안전망으로 계속 유지하며, 이 명령이 건드리지 않는다. 향후 게이트가 "이미 명시적으로 리뷰된 세션"을 인식해 중복을 건너뛸 수 있는 설정이 생기면 그때 별도로 검토한다.

## 최종 보고 형식

1. 설계 리뷰에서 자동 수정한 사실 오류 목록
2. 자동 결정한 구현 세부사항과 근거
3. `KNOWN_ISSUES.md`에 기록한 항목
4. `TODO.md`에 추가한 아이디어
5. 사용자 확인이 실제로 발생했다면 그 시점과 결과
6. 커밋 전 리뷰 반복 횟수와 최종 결과(2회 초과로 중단됐다면 그 사실과 남은 지적 사항)
7. 커밋 해시, 푸시 결과, 배포 확인 결과
8. 이번 실행에서 `stop-review-gate`가 중복 발동했는지 여부
