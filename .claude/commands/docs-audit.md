# /docs-audit — 문서 정합성 전체 점검 및 자동 수정

매번 전체 문서를 다시 훑는 무거운 작업이다. 매 세션마다 쓰기보다는, 큰 규칙 변경이 몇 번 누적된 뒤 한 번씩 돌리는 용도로 쓴다.

이 명령이 호출되면 아래 절차를 순서대로 수행한다.

## 1단계 — 점검

다음 파일 전체를 읽고 서로 같은 규칙/사실을 설명하는 문장들을 대조한다:
- 루트: CLAUDE.md, AGENTS.md
- docs/ 전체
- .claude/ 전체 (rules/, agents/, commands/, output-styles/, settings.json, hooks/*.sh)
- .github/copilot-instructions.md

중점 확인 항목:
- SQL/DB 실행 위험도 규칙
- Codex 리뷰 절차(stop-review-gate)
- Mode A/B 워크플로우 순서
- git commit/push 순서
- 존재하지 않는 파일/함수 참조
- index.html 실제 코드와 문서 설명의 일치 여부 (함수명, RLS 정책, 데이터 흐름 등 — grep과 Supabase MCP로 직접 검증, 추측 금지)

## 2단계 — 분류

발견한 불일치를 두 그룹으로 나눈다.

**그룹 A (바로 수정)**: 아래 조건을 모두 만족하는 것만
- 코드/DB를 직접 확인해서 정답이 명확히 하나로 확정되는 것 (예: 존재하지 않는 함수명, 실제로는 구현된 기능을 "없음"이라고 쓴 경우, 날짜가 명시된 최신 규칙과 안 맞는 옛 문구)
- 여러 문서에 같은 사실이 다르게 적혀 있고, 그중 한쪽이 날짜/코드 근거로 명백히 최신인 경우

**그룹 B (사용자 확인 필요, 수정하지 않음)**: 코드/문서만으로는 정답을 확정할 수 없는 것
- 실행 환경에 따라 동작이 달라 보이는 것 (예: 설정은 정상인데 특정 환경에서만 동작이 다르게 관찰되는 경우)
- 의도적 설계인지 누락인지 애매한 것
- 두 문서가 각자 다른 맥락에서 쓴 것일 수 있는 것

## 3단계 — 그룹 A 처리

그룹 A는 하나씩 수정하고, CLAUDE_CODE_RULES.md의 "커밋 전 필수 리뷰 규칙"에 따라 Codex 리뷰를 거친 뒤 **모든 그룹 A 수정을 모아 한 번에 커밋**한다 (파일 하나당 커밋 여러 개로 쪼개지 않는다). 리뷰 중 범위 밖 이슈가 추가로 발견되면 임의 수정하지 않고 그룹 B로 옮겨 보고한다.

## 4단계 — 보고

다음 형식으로 최종 보고한다:
1. 그룹 A — 수정 완료 목록 (파일:줄, 수정 전/후, 근거)
2. 그룹 B — 사용자 확인 필요 목록 (무엇이 애매한지, 확인하려면 어떤 정보/실행이 필요한지)
3. 커밋 여부 및 커밋 메시지

## 원칙

- 그룹 B는 절대 임의로 수정하지 않는다.
- 코드/DB 확인 없이 "아마 이게 맞을 것 같다"는 추측으로 그룹 A에 넣지 않는다.
- MAINTENANCE.md 기준으로 이 점검·수정이 다른 문서(TODO/CHANGELOG/DECISIONS 등) 갱신을 유발하는지 마지막에 확인한다.
